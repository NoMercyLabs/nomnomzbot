# Frontend / UI — Design (DRAFT — SUPERSEDED)

> **SUPERSEDED (2026-06-17) by `spec/frontend.md`.** This early draft is kept for history only. It is **stale on one major point**: it argued for "no web dashboard" to sidestep Compose-for-Web risk. That was reversed in the stack reconsideration — the current binding position (CLAUDE.md + `spec/frontend.md`) is that **desktop and web (wasmJs) ship the identical first-class dashboard from one codebase**. For the authoritative stack, navigation/state/structure, the typed client (REST + hand-rolled SignalR), the connection/profile model, i18n, and design-system mapping, read **`spec/frontend.md`**.

Source: design dialogue 2026-06-16.

## Stack
- **Kotlin Multiplatform (KMP) + Compose Multiplatform** — shared logic AND shared Compose UI. Desktop first, mobile (Android/iOS) later. Replaces the deleted React Native app (Stoney dislikes React; KMP/Compose avoids it).

## The web-risk solve: desktop is the universal client
- The **dashboard is the KMP + Compose desktop app** (and later mobile) — and it's **profile-agnostic**: it just needs a backend URL.
  - Self-host → points at `localhost`.
  - SaaS → points at the SaaS API.
- A SaaS user **installs the desktop app and connects to the SaaS** — **no web dashboard needed**, which **sidesteps the Compose-for-Web/Wasm maturity risk** entirely.
- **Web is only for the simple public pages** viewers/OBS hit without installing anything:
  - Song-request page (viewers)
  - Overlays / widgets (OBS browser source — the widget system)
  - OAuth callback landing

  Kept lightweight (plain / Svelte / server-rendered) — not the heavy dashboard, so no Compose-web question.

## Backend comms
- Typed shared KMP client: **REST** (the v1 API) + **SignalR** for realtime (dashboard / overlay hubs).
