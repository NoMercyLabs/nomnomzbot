# Frontend Structure — the placement rulebook

**Area:** where every frontend artifact lives, how it is registered, and the line between the engine
(`core/`) and feature code. There is exactly **one** home for each thing — never "it could also go
here." This is the rulebook every contributor and scaffold follows, the sibling of
`backend-structure.md`.

**Applies to:** `app/composeApp/src` (the one KMP module). Backend layout is owned by
`backend-structure.md`; the design layer by `frontend-design-system.md`; the query engine by
`frontend-data-layer.md`.

**Conventions:** `commonMain`-first with full `wasmJs` parity; explicit types; one public type per
file; **package == folder path**; AGPL header on every `.kt`; `ApiResult<T>` over exceptions/null;
async via coroutines (never block). Package root is `bot.nomnomz.dashboard`.

---

## 0. Decisions (binding)

- **F1 — One module, thin platforms.** A single `:composeApp` with targets `jvm` + `wasmJs`; platform
  source sets hold only `actual` seams. All UI, state, data, and engine live in `commonMain`. (Per
  `frontend.md` §1; a `:shared` module is added only on Rule-of-Three.)
- **F2 — Two zones, legible by location.** `core/` is the **engine** (network, realtime, query,
  connection, designsystem, navigation, di, i18n) — rarely changes. `feature/<x>/` is **user-facing**.
  The split is the path; no markers, no separate module. (Mirrors backend D4.)
- **F3 — Feature-first packages.** Organize by feature/domain, never by provenance. **Banned folder
  names:** `utils`, `helpers`, `misc`, `common`, `shared` (as a dump), `generated` (except the single
  committed OpenAPI folder, F6). (Mirrors backend D2.)
- **F4 — Closed artifact taxonomy (§2).** Every `.kt` is one of the listed types; each has exactly one
  folder pattern and one base type/marker. Learn it once → you always know where a thing goes.
  (Mirrors backend D3.)
- **F5 — Deterministic registration, no forgotten wires.** Every feature exposes one Koin module; a
  single committed aggregate lists them; the **linter red-builds if a `*Module` (or a `Route`) is not
  in the aggregate**. That achieves backend D5's *outcome* ("you cannot ship an unregistered artifact") by a
  different, **wasm-safe mechanism** (hand aggregate + linter — assembly-scan/reflection is unavailable
  on wasmJs); it upgrades to Koin annotation-scan 1:1 when that path is stable (§4).
- **F6 — Generated contracts: committed, never hand-edited.** The v1 OpenAPI generator emits REST
  DTOs + raw stubs into the **one** allowed `core/network/generated/` folder (`// <auto-generated />`
  first line), regenerated on contract change, hand-wrapped by facades. SignalR has no schema → hand
  authored. (Per `frontend.md` §3; mirrors the backend's generated-Helix rule.)
- **F7 — One of each engine singleton.** Exactly one `HttpClient`, one `QueryClient`, one realtime
  client, one `ConnectionStore`. Duplicates are a linter failure. (No backend D-number twin — backend's "one of each" falls out of its D5 scan; here it is a standalone linter rule.)
- **F8 — Scaffolds, not improvisation.** A template per artifact type drops a new file in its
  canonical home with header, base type, and registration stub pre-filled (§6). (Mirrors backend D8.)

## 1. The module & its zones

```
app/composeApp/src/
├─ commonMain/kotlin/bot/nomnomz/dashboard/
│  ├─ App.kt                      # theme + connection gate + NavHost root
│  ├─ core/                       # ENGINE (F2) — rarely changes
│  │  ├─ network/                 # HttpClient config, ApiResult, auth/refresh, per-subsystem facades
│  │  │  └─ generated/            # OpenAPI DTOs + raw stubs (committed, auto-generated — F6)
│  │  ├─ realtime/                # hand-rolled SignalR client + typed hub clients
│  │  ├─ query/                   # the QueryClient engine (frontend-data-layer.md)
│  │  ├─ connection/              # ConnectionProfile/Store, mDNS (expect), TokenVault (expect)
│  │  ├─ designsystem/            # theme/ (tokens) + color/ + component/ + pattern/ + icon/ (frontend-design-system.md)
│  │  ├─ navigation/              # Route graph, shell, gate
│  │  ├─ i18n/                    # locale state + Res helpers
│  │  └─ di/                      # coreModule + AppModules aggregate
│  └─ feature/<x>/                # USER-FACING (F2) — one folder per feature
│     ├─ ui/                      # composables (screens, sections, rows)
│     ├─ state/                   # plain state-holders + feature Stores (no androidx ViewModel)
│     ├─ data/                    # query/mutation hooks, query keys, facade wiring
│     └─ di/                      # the feature's Koin module
├─ jvmMain/…                      # actuals: CIO engine, OS-vault, JmDNS, loopback OAuth, Swing dispatcher
└─ wasmJsMain/…                   # actuals: JS engine, sessionStorage token, no-op discovery, redirect OAuth
```

Features (mirroring the backend modules + `frontend.md` screens): `setup` · `dashboard` · `commands`
· `pipeline` · `community` · `moderation` · `rewards` · `timers` · `widgets` · `integrations` ·
`settings`.

## 2. The artifact taxonomy

Every `.kt` is one of these. `Folder` is relative to the package root; `<x>` is a feature.

| Artifact | Zone | Folder | Base type / marker | Registered |
|---|---|---|---|---|
| Screen | feature | `feature/<x>/ui/` | `@Composable …Screen(...)` | nav graph (§4) |
| Section / row composable | feature | `feature/<x>/ui/` | `@Composable` | — |
| State-holder (presenter) | feature | `feature/<x>/state/` | plain `class …State` (StateFlow) — **not** androidx ViewModel | feature module if injected |
| Store (global state) | core/feature | `core/<area>/` or `feature/<x>/state/` | `class …Store` (StateFlow singleton) | module (single) |
| Query hook | feature | `feature/<x>/data/` | `@Composable fun use…(): Query<T>` | — (calls QueryClient) |
| Mutation hook | feature | `feature/<x>/data/` | `@Composable fun use…Mutation(): Mutation<…>` | — |
| Query keys | feature | `feature/<x>/data/` | `object …Keys` | — |
| REST facade | core | `core/network/<x>/` | `class …Api` | core/feature module |
| Generated DTO/stub | core | `core/network/generated/` | `@Serializable` (auto-gen) | — |
| Hub client | core | `core/realtime/` | `interface …HubClient` + `internal class Default…HubClient` (separate files) | core module |
| DS primitive | core | `core/designsystem/component/` | `@Composable` + `*Styles.resolve()` + variant/size enums, row in `catalogue.md` | — |
| DS pattern | core | `core/designsystem/pattern/` | `@Composable` | — |
| Theme / token / color util | core | `core/designsystem/theme/` (tokens + theme) · `core/designsystem/color/` (OKLCH util) | — | — |
| Icon set | core | `core/designsystem/icon/` | `: IconSet` | core module |
| Route | core | `core/navigation/` | `@Serializable … : Route` | nav graph (§4) |
| Nav graph contribution | feature | `feature/<x>/ui/` | `fun NavGraphBuilder.<x>Graph(...)` | shell (§4) |
| String resource | core | `composeResources/values*/` | xml (`en` + `nl`) | — |
| DI module | core/feature | `core/di/` or `feature/<x>/di/` | `val …Module: Module` | **AppModules aggregate (linted, F5)** |
| expect/actual seam | core + platform | `core/<area>/` (expect) + `{jvm,wasmJs}Main` (actual) | `expect` / `actual` | — |

> A **state-holder** (feature/state, holds screen logic) is distinct from a **Store** (global,
> long-lived, injected) and from a **query hook** (reads server state from the cache). Three different
> homes because they have three different lifetimes — exactly the §4 state model of `frontend.md`.
>
> **Single-home rule for the dual-zone rows.** A `Store` / `DI module` / `expect`-`actual` seam is
> `core/` **iff** it belongs to an engine area (the F2 list: network, realtime, query, connection,
> designsystem, navigation, di, i18n); otherwise it is `feature/<x>/` — every feature owns its own
> `feature/<x>/di/` module and `feature/<x>/state/` stores. No real choice in practice.

## 3. Feature anatomy — worked example (Commands)

```
feature/commands/
  ui/      CommandsScreen.kt · CommandRow.kt · CommandEditorSheet.kt
  state/   CommandEditorState.kt          # plain holder: edit buffer, validation, submit()
  data/    CommandsApi.kt (facade) · useCommands.kt · useCommandMutations.kt · CommandsKeys.kt
  di/      commandsModule.kt              # val commandsModule: Module = module { … }
```

- `useCommands()` calls `useQuery(CommandsKeys.list(channelId)) { commandsApi.list(...) }` (which binds
  the injected `QueryClient.observe`) and returns a `Query<List<CommandDto>>`; the screen renders its
  `isLoading/data/error`.
- `useCommandMutations()` exposes `create/update/toggle/delete` mutations that invalidate
  `CommandsKeys.list(...)` on success (optimistic where it helps).
- **Query-key space mirrors the backend routes** — `CommandsKeys` tracks `CommandsController`, so an
  invalidation pushed over SignalR maps to the same key the screen reads.

## 4. Registration (F5) — the determinism guarantee

- Each feature exposes `val <x>Module: Module`. `core/di/AppModules.kt` holds the **one** aggregate:
  `val appModules = listOf(coreModule, designSystemModule, commandsModule, /* … */)`.
- Started once per platform entry: `initKoin { modules(appModules) }`.
- **The linter fails the build if** a `*Module` exists that is **not** in `appModules` → forgetting to
  register is impossible (backend D5's *outcome*, achieved without reflection — unavailable on wasmJs).
- **Navigation registers the same way.** Each feature contributes `fun NavGraphBuilder.<x>Graph()` and
  a `Route`; the shell's single `NavHost` calls each `…Graph()`; the linter fails the build if **a `*Graph()`
  extension is never invoked by the shell, or a `Route` is declared but never wired**. One NavHost, no
  orphan routes or graphs.
- **Upgrade path:** when Koin's compiler-plugin annotation-scan is stable + wasm-documented, `@Module`
  / `@ComponentScan` replace the hand aggregate 1:1 and the "in AppModules" lint rule retires — no
  structural change to features.

## 5. The generated layer (F6)

One Gradle `openapi-generator` task (`generatorName=kotlin`, `library=multiplatform`) runs against the
backend's published v1 OpenAPI document → Kotlin `@Serializable` DTOs + raw endpoint stubs in
`core/network/generated/`, committed, `// <auto-generated />` first line, **never hand-edited**.
Per-subsystem `…Api` facades wrap the stubs and return `ApiResult<T>` (mirrors the backend's
generated-Helix-client rule: external contracts are generated, not transcribed).

## 6. Scaffolds (F8)

A template per artifact type, each emitting files in their canonical homes with header + base + wiring
stub:

| Template | Emits |
|---|---|
| `nomnomz-feature <x>` | `feature/<x>/{ui,state,data,di}/` skeleton + `<x>Module` + `<x>Graph()` + **inserts `<x>Module` into `core/di/AppModules.kt`** (the linter is the backstop, F5) |
| `nomnomz-screen <x> <Name>` | `feature/<x>/ui/<Name>Screen.kt` + `Route` + graph entry |
| `nomnomz-query-hook <x> <Name>` | `use<Name>.kt` + `<x>Keys` entry + facade method stub |
| `nomnomz-component <Name>` | `core/designsystem/component/<Name>.kt` + variant enums + resolver + `catalogue.md` row |
| `nomnomz-store <Name>` | `…Store.kt` (StateFlow) + module binding |

## 7. Linting — the rulebook is enforced

A detekt ruleset (+ CI gate), the frontend `taxonomy-linter`, fails the build on:

- a **banned dump folder** (F3), or an artifact whose folder doesn't match its base type (§2);
- a **package that doesn't equal its folder path** (hand-written `.kt` under `kotlin/` only; the OpenAPI generator emits `packageName = bot.nomnomz.dashboard.core.network.generated`, and `composeResources/` XML is exempt), or a missing AGPL header;
- a `*Module` **not** in `AppModules`, or a `Route` **not** wired into the shell (F5);
- more than one `HttpClient` / `QueryClient` / realtime client (F7);
- a hand-edit inside `core/network/generated/`;
- plus the **design-system rules** (`frontend-design-system.md` §8): raw color/`dp`, off-catalogue
  components, direct `MaterialTheme.*` reads, hardcoded user-facing strings, raw icon paths.

The `app/` KMP module is greenfield. The first vertical slice (per `frontend.md` and the

backend — proving module layout, both targets, DI, the generated REST path, and runtime config in one
commit. The QueryClient engine and navigation land in the next slices, on the proven spine.

## 9. Decisions (resolved)

All settled and binding:
- One `:composeApp` module, `jvm` + `wasmJs`, thin platforms (F1).
- Two zones by location: `core/` engine vs `feature/<x>/` (F2); feature-first, banned dump folders (F3).
- Closed artifact taxonomy, one home per type (F4, §2).
- Deterministic registration: per-feature Koin module + linter-enforced aggregate, same for routes
  (F5, §4); annotation-scan upgrade later.
- Generated REST contracts committed + never hand-edited (F6, §5); hand-authored SignalR.
- One of each engine singleton (F7); scaffolds per artifact type (F8, §6).
- Everything enforced by a detekt `taxonomy-linter` + CI gate (§7).
